\section{Approach}\label{sec:approach}

\subsection{Overview}\label{sec:overview}

We propose an LLM-driven pipeline for recovering trace links between
Software Architecture Documents (SAD) and Software Architecture Models (SAM).
Given a natural-language architecture document~$D$ consisting of sentences
$\{s_1, \ldots, s_n\}$ and a component model~$M$ containing components
$\{c_1, \ldots, c_m\}$, our approach produces a set of trace links
$L \subseteq S \times C$, where each link $(s_i, c_j)$ asserts that
sentence~$s_i$ discusses component~$c_j$.

\paragraph{Challenge.}
Architecture documents refer to components in diverse ways:
by canonical name, by abbreviation, by functional description,
by trailing word of a compound name (e.g., ``Dispatcher'' for
\texttt{EventDispatcher}), or implicitly via pronouns.
No single extraction strategy handles all reference types
well---a high-precision name matcher misses aliases and pronouns,
while a broad semantic extractor introduces false positives from
coincidental word overlap.

\paragraph{Key insight.}
Different reference types require different resolution
strategies, but these strategies can operate independently as long
as they share a common knowledge base.
This suggests a fan-out architecture where each strategy addresses
exactly one reference type, contributing candidates additively without
interfering with each other's precision/recall trade-offs.

\paragraph{Design principles.}
Three principles guide the pipeline design:
\begin{itemize}
    \item \emph{Multi-pass consensus:}
      Precision-critical decisions use intersection voting---two
      independent LLM passes must agree---to suppress hallucinations
      and LLM non-determinism (\S\ref{sec:seed},
      \S\ref{sec:entity}, \S\ref{sec:patterns}).
    \item \emph{Citation-grounded verification:}
      Rather than trusting LLM judgments as opaque decisions, we
      require the LLM to produce verifiable evidence (antecedent
      citations, exact substrings) that is checked deterministically
      in code (\S\ref{sec:coref}, \S\ref{sec:entity}).
    \item \emph{Benchmark-clean prompts:}
      All few-shot examples use abstract placeholders or safe
      textbook domains; no benchmark names or gold-standard
      information appears in any prompt.
\end{itemize}

\paragraph{Architecture.}
The pipeline is organized as a staged DAG
(Figure~\ref{fig:pipeline}) with three layers:

\begin{enumerate}
    \item \textbf{Knowledge acquisition} (Layer~1): Independent
      analyses extract structured knowledge from the raw inputs---model
      structure, document vocabulary, and linguistic patterns.
    \item \textbf{Link recovery} (Layer~2): Four complementary
      strategies produce candidate links, each targeting a distinct
      reference type (\S\ref{sec:recovery}). Their outputs converge
      only at the merge point.
    \item \textbf{Link consolidation} (Layer~3): A single fan-in
      stage merges all candidates, applies convention-aware boundary
      filtering to \emph{all} links regardless of provenance, and
      removes residual false positives via a deterministic evidence
      filter.
\end{enumerate}

\noindent Explicit reference extraction is logically a Layer~2
strategy (it produces links, not knowledge), but because it requires
no Layer~1 knowledge, it runs concurrently with Layer~1 to overlap
its LLM cost with the knowledge phases.

\subsection{Knowledge Acquisition}\label{sec:knowledge}

Layer~1 builds a shared knowledge base from the raw document and model.
Model analysis, document analysis, and explicit reference extraction
are mutually independent---each reads only the raw inputs---enabling
concurrent execution.
Pattern learning depends on model analysis results and runs once
the initial analyses complete.

\subsubsection{Model Analysis}\label{sec:model-analysis}

Component names vary widely in specificity.
Constructed identifiers like \texttt{TaskScheduler} unambiguously name
architectural roles, but single common-English words like \texttt{Core}
or \texttt{Adapter} appear frequently in non-referential contexts.
To distinguish them, we classify each name as \emph{architectural}
or \emph{ambiguous} using a two-step hybrid:
a few-shot LLM prompt makes an initial classification, then a
deterministic structural guard overrides the LLM for names with
unambiguous structure---multi-word names, CamelCase compounds, and
all-uppercase acronyms are always classified as architectural.
This ensures LLM judgment is used only where structural cues are absent.

The analysis additionally extracts \emph{implementation relationships}
between component names
(e.g., \texttt{BasicScheduler}~$\to$~\texttt{Scheduler}),
identified by shared substrings, and \emph{generic partial words}---the
set of ambiguous words that also appear as subwords of other component
names---used later to calibrate abbreviated reference matching
(\S\ref{sec:partial}).

\subsubsection{Document Analysis}\label{sec:doc-analysis}

Architecture documents rarely refer to components exclusively by their
canonical model names.
Authors abbreviate (\emph{AST} for \texttt{AbstractSyntaxTree}),
use trailing words as shorthand (\emph{Dispatcher} for
\texttt{EventDispatcher}), or introduce synonyms (\emph{task runner}
for \texttt{TaskScheduler}).
Without an \emph{alias table}---a mapping from alternative names to
canonical components---any name-matching strategy will systematically
miss these references, capping recall.

\paragraph{Alias discovery.}
An LLM extracts three categories of aliases from the document:
\begin{itemize}
    \item \emph{Abbreviations}: short forms via parenthetical
      definitions (e.g., ``Abstract Syntax Tree (AST)'').
    \item \emph{Synonyms}: alternative names that unambiguously
      identify exactly one component.
    \item \emph{Partial references}: trailing words of multi-word
      component names used alone (e.g., ``Dispatcher'' for
      \texttt{EventDispatcher}).
\end{itemize}

\noindent A calibrated few-shot judge validates each proposed mapping,
auto-approving abbreviations formed from component-name initials,
trailing words of multi-word names, and CamelCase identifiers, while
rejecting generic terms (\emph{system}, \emph{process}).
A deterministic \emph{CamelCase rescue} overrides any judge rejection
of CamelCase identifiers, since constructed identifiers are never
generic English words.

\paragraph{Partial-reference refinement.}
Among the three alias types, partial references require the most
careful handling: trailing-word shorthands are pervasive in
architecture documents, but many trailing words are ordinary English
words whose referential status depends on context.
A three-step pipeline progressively refines which partials are safe:
\begin{enumerate}
    \item \emph{Statistical enrichment:}
      If a trailing word appears in $\ge3$ sentences where the full
      name is absent and no other component shares that word, the
      partial mapping is added automatically.
      Ambiguous trailing words require capitalized occurrences.
    \item \emph{Usage classification:}
      Per-partial LLM calls determine whether each generic partial
      is typically used as a component reference (\textsc{name}) or
      as an ordinary English word (\textsc{ordinary}).
    \item \emph{Downstream gating:}
      \textsc{Ordinary}-classified partials remain in the alias table
      for abbreviated reference matching (\S\ref{sec:partial})
      but are excluded from the confirmed-alias context provided to
      the boundary filter (\S\ref{sec:boundary}), preventing it from
      treating generic uses as validated references.
\end{enumerate}

\subsubsection{Pattern Learning}\label{sec:patterns}

Architecture documents describe not only components but also their
internal parts---subprocess terms like ``garbage collector'' inside a
\texttt{MemoryManager}.
If left unrecognized, these terms cause false positive links in
downstream validation and anaphoric resolution, where a sentence about
a subprocess is misattributed to the enclosing component.

Two independent LLM passes identify subprocess terms via a
cross-validation protocol:
the first pass proposes candidates; the second independently
validates them, rejecting terms that may be valid component references.
Only terms confirmed by both passes are retained.
Subprocess terms serve as exclusion criteria in contextual validation
(\S\ref{sec:entity}) and anaphoric resolution (\S\ref{sec:coref}).

Pattern learning reads the filtered component list (excluding
implementation names) and the document sentences.
This dependency on model analysis places it after the initial
knowledge acquisition, forming a brief enrichment step before
link recovery begins.

\subsection{Link Recovery}\label{sec:recovery}

Layer~2 comprises four complementary strategies, each targeting a
distinct reference type:

\begin{itemize}
    \item \textbf{Explicit reference extraction} (\S\ref{sec:seed})
      recovers verbatim component names using only raw inputs,
      optimizing for precision.
    \item \textbf{Contextual reference discovery} (\S\ref{sec:entity})
      uses the full alias table to recover references requiring
      contextual understanding, then validates candidates through
      a multi-step pipeline.
    \item \textbf{Anaphoric reference resolution} (\S\ref{sec:coref})
      resolves pronominal and definite-description references
      back to previously mentioned components.
    \item \textbf{Abbreviated reference matching} (\S\ref{sec:partial})
      deterministically matches partial names from the alias table.
\end{itemize}

\subsubsection{Explicit Reference Extraction}\label{sec:seed}

Explicit extraction establishes a high-precision baseline
\emph{before} any knowledge is available.
It runs a self-contained LLM-based extractor (\textbf{ILinker2})
that reads only the raw sentences and component list---no
abbreviations, synonyms, or partial references---enabling concurrent
execution with Layer~1 knowledge acquisition.

ILinker2 uses two complementary LLM passes over batched document
segments, each with a different prompt framing:

\begin{itemize}
    \item \textbf{Pass~A (mention-framed):} ``For each sentence,
      identify which components are explicitly mentioned.''
    \item \textbf{Pass~B (actor-framed):} ``For each sentence,
      determine which component is the subject or primary actor.''
\end{itemize}

\noindent Both passes reject names appearing only inside dotted
package paths (e.g., \texttt{renderer.utils.config} does not
reference \texttt{Renderer}) and generic English words used in
their ordinary sense.

The merge strategy applies \emph{asymmetric voting}:
exact-name matches from \emph{either} pass are accepted (union),
while non-exact matches require agreement from \emph{both} passes
(intersection).
ILinker2 is designed for high precision at the cost of recall;
the recall gap---alias-based references, functional descriptions,
pronominal coreferences---is addressed by the three knowledge-enriched
strategies that follow.

\subsubsection{Contextual Reference Discovery}\label{sec:entity}

Where explicit extraction finds only references recognizable from the
component name alone, contextual discovery leverages the full alias
table to recover references that \emph{require} contextual
understanding.
It differs from explicit extraction in three ways:

\begin{enumerate}
    \item \textbf{Knowledge-enriched prompts.}
      The discovery prompt receives all abbreviations, synonyms, and
      partial references from the alias table, enabling it to
      recognize aliases (e.g., ``AST'' for \texttt{AbstractSyntaxTree})
      that the knowledge-free explicit extractor cannot.
    \item \textbf{Broader inclusion scope.}
      Beyond direct name matches, the prompt covers additional
      reference patterns: space-separated forms of compound names,
      functional descriptions, known aliases, interaction participants
      (``X sends data to Y'' references both X and Y), and
      passive/prepositional mentions (``handled by X'').
    \item \textbf{Same-prompt intersection.}
      Contextual discovery runs the \emph{same} prompt \emph{twice}
      independently and keeps only candidates found in \emph{both}
      passes.
      This eliminates spurious candidates from LLM non-determinism---a
      risk that grows with the broader inclusion scope.
\end{enumerate}

For components that receive no links from either explicit extraction
or the discovery phase, a \emph{targeted recovery} step issues
single-component prompts with all known aliases, recovering references
missed by broad-scope batch processing.

\paragraph{Validation.}
Candidates from contextual discovery undergo a three-step validation
pipeline:

\begin{enumerate}
    \item \emph{Generic-mention detection.}
      For candidates involving an ambiguous component name in lowercase
      (e.g., ``core'' rather than ``Core''), an LLM determines whether
      the usage refers to the architectural component or is an ordinary
      English word.
      The LLM sees anchor sentences where the full name appears for
      calibration, and classifies each candidate as \textsc{component}
      or \textsc{generic}.
      Generic-classified candidates are rejected before further
      validation.
    \item \emph{Code-first auto-approval.}
      Candidates where a component name or alias appears verbatim
      in the sentence text (verified by word-boundary matching in
      code) are approved without LLM cost.
    \item \emph{Two-pass intersection.}
      Remaining candidates are validated by two independent LLM passes
      with complementary foci---one on \emph{actor role}, one on
      \emph{direct reference}.
      A candidate is accepted only if both passes approve it.
\end{enumerate}

\subsubsection{Anaphoric Reference Resolution}\label{sec:coref}

Sentences containing pronouns (\emph{it}, \emph{they}, \emph{this},
\emph{these}, \emph{that}, \emph{those}) and definite descriptions
(\emph{the component}, \emph{the service}) carry valid trace
information but contain no component name, making them invisible to
explicit extraction and contextual discovery.
However, the component a pronoun refers to was typically named
explicitly in a nearby sentence.
If the LLM can identify this \emph{antecedent}---and the antecedent
can be verified deterministically---the resolution becomes trustworthy
without additional review.

The pipeline uses a unified \emph{cases-in-context} strategy:
each pronoun-bearing sentence is presented as an individual case with
a $\pm5$-sentence bidirectional context window, processed in batches
of~10.
The LLM sees each case's surrounding context (marked with
\texttt{>>>} for the target sentence) and resolves any pronoun
that refers to an architecture component.
This per-case presentation is robust across both simple and complex
documents, eliminating the need for document-complexity gating.

The resolution enforces citation-grounded verification:
the LLM must cite an antecedent sentence where the component was
explicitly named, along with the exact quote containing the name.
This citation is verified deterministically:
\begin{enumerate}
    \item The component name (or a known alias) must appear verbatim in
      the cited antecedent sentence.
    \item The antecedent must be within 3~sentences of the pronoun.
    \item Sentences identified as subprocess descriptions by pattern
      learning are excluded.
\end{enumerate}

\noindent This hybrid LLM-plus-code verification prevents hallucinated
resolutions where the LLM fabricates a nonexistent antecedent.
Because the verification is strict (verbatim name match within
3~sentences), the code-level check effectively guards against false
positives, making anaphoric links safe to accept without further review
(validated empirically in \S\ref{sec:evaluation}).

\subsubsection{Abbreviated Reference Matching}\label{sec:partial}

Contextual discovery may miss sentences where a partial name appears
without surrounding semantic cues.
Determining whether a partial name like ``Server'' refers to the
component or is used generically requires project-specific context
that a single LLM prompt cannot reliably encode.

Instead, the pipeline deterministically scans all sentences for
word-boundary matches of each partial reference from the alias table,
producing links for pairs not already covered by preceding strategies.
A \emph{clean mention} filter ensures the partial name appears as a
standalone word---not inside a dotted path, a hyphenated compound,
or adjacent to a period indicating a qualified name.
This strategy uses no LLM calls, providing stable recall at zero
additional inference cost.
Its output is evaluated individually by the boundary filter in
Layer~3 (\S\ref{sec:boundary}).

\subsection{Link Consolidation}\label{sec:merge}

Layer~3 merges the outputs of all recovery strategies and applies
two filtering stages to remove false positives.

\subsubsection{Priority-Based Merge}\label{sec:dedup}

The merge stage deduplicates links by $(s_i, c_j)$ pair.
When multiple strategies produce the same link, the highest-priority
source is retained:
$\text{explicit} > \text{contextual} > \text{anaphoric} >
\text{abbreviated}$,
reflecting decreasing levels of direct textual evidence.
A \emph{parent-overlap guard} removes links to implementation
components when the abstract parent is already linked to the same
sentence (e.g., if both \texttt{Scheduler} and
\texttt{BasicScheduler} are linked, the implementation variant is
dropped).

\subsubsection{Convention-Aware Boundary Filter}\label{sec:boundary}

Even after merge, some links survive where a component name appears
in a sentence but is used in a non-architectural sense.
A structured LLM filter applies a three-step reasoning guide to
\emph{all} links, regardless of provenance:

\begin{enumerate}
    \item \textbf{Hierarchical name reference:}
      Reject if the component name appears only as a prefix in a
      dotted or qualified path (e.g., ``\texttt{X.utils} provides
      helper functions'' does not reference component~X).
      An exception preserves links when the sentence also names
      the component explicitly.
    \item \textbf{Name confusion:}
      Reject if the name is used as a technology, methodology, or
      generic English word in a non-architectural sense.
      A critical exception preserves links when a component is
      \emph{named after} a technology---any sentence describing that
      technology's capabilities is about the component.
    \item \textbf{Default:}
      If neither rejection pattern applies, the link is kept.
\end{enumerate}

\noindent Structural guardrails ensure multi-word component names and
CamelCase identifiers are never treated as generic words, and that
interaction sentences produce links for all participating components.

\subsubsection{Evidence Filter}\label{sec:evidence-filter}

A lightweight deterministic filter removes links that lack textual
evidence.
A link $(s_i, c_j)$ is retained if \emph{any} of the following holds:

\begin{enumerate}
    \item The component name appears as a standalone word-boundary
      match in the sentence text, excluding occurrences inside dotted
      paths or hyphenated compounds.
    \item A known alias from the alias table appears in the sentence.
    \item The link originates from explicit extraction or anaphoric
      resolution, both of which already verified the reference at
      their respective extraction stages.
\end{enumerate}

\noindent Links satisfying none of these conditions---where neither
the component name, an alias, nor prior verification provides
evidence---are dropped.
This filter requires no LLM calls and catches residual false
positives where contextual discovery hallucinated a reference that
does not appear in the text.
